Kite Candles

Streams live tick data from the Zerodha Kite WebSocket, fans it through Redis Pub/Sub, and assembles OHLCV candle bars in real time — one producer feeding unlimited timeframe consumers.

Overview

Data Source
Kite WebSocket (Full mode)
Message Bus
Redis Pub/Sub
Output
Multi-timeframe OHLCV bars → CSV + Redis channels
Language
Python 3 — threading, Queue, redis-py

The system is split into two independently deployable processes connected by a Redis channel. The producer owns the WebSocket connection and publishes every tick batch as JSON. Each consumer (one per timeframe) subscribes to that channel and builds its own candles, writing completed bars to both Redis and a CSV file.

Kite WebSocket
→
kite_ticker_producer.py
→
Redis: ticks

1min · 5min · 15min · 30min · 1h + CSV

Module Descriptions

set_token.py one-time daily

Stores today's Kite session token in Redis so all other processes can read it without touching the filesystem. Edit the two token constants at the top, then run once after completing the Kite login flow.

DetailValue
Reads from~/kite_candles/userdata
Writes toRedis hash token.{user_id}
Fieldsaccess_token, public_token
When to runOnce each morning after Kite login

instruments_generator.py daily cron

Downloads the full NSE/NFO instrument master from the Kite REST API. The CSV maps instrument_token integers (used in WebSocket ticks) to human-readable tradingsymbol strings. Both the producer and consumer read this file at startup.

DetailValue
EndpointGET /instruments
Output~/kite_candles/instruments.csv
Retry behaviourPolls every 15 s until response > 2 048 bytes
When to runDaily before market open (crontab 06:00)

kite_ticker_producer.py long-running

Maintains a persistent WebSocket connection to Kite, receives every market tick in real time, and fans each batch out to two Redis destinations simultaneously.

Startup sequence

  1. Reads userdata for api_key and user_id.
  2. Reads access_token from Redis hash token.{user_id}.
  3. Starts 4 publish worker threads all draining a single publish_queue.
  4. Starts 1 tick-hash worker thread draining tick_hash_queue.
  5. Opens the Kite WebSocket and registers all callbacks.

on_connect — instrument subscription

  • Computes the nearest weekly options expiry and front futures month.
  • Builds a combined universe: NIFTY/BANKNIFTY options + front-month futures + NSE cash equities + NIFTY 50 / NIFTY BANK / INDIA VIX indices.
  • Subscribes all tokens in MODE_FULL (full depth + OHLC + volume).
  • Builds allinstrumentMap covering the entire instrument master.

on_ticks — per-batch callback

Runs on the WebSocket thread — must be fast with no I/O. Puts the entire batch into publish_queue and each individual tick into tick_hash_queue. Calls os._exit(0) if invoked outside 09:00–15:30.

_publish_worker (×4 threads)

Serialises each tick batch to JSON with a custom _DatetimeEncoder (converts datetime/date objects to ISO-8601 strings), then publishes to Redis channel ticks. Four threads share one queue — if one publish blocks on network, the others continue.

_tick_hash_worker (×1 thread)

Writes per-symbol real-time price fields into dedicated Redis hashes, enabling downstream consumers to query the last price of any symbol without processing the full tick stream.

multi-time/1min/min1_ticks.py long-running

Subscribes to the ticks Redis channel and builds OHLCV candles for every configured timeframe simultaneously — one Redis subscription, any number of independent candle widths, in a single process.

Startup sequence

  1. Reads instruments.csv → builds allinstrumentMap and tracked_symbols.
  2. Creates one Queue per timeframe in tf_queues.
  3. Subscribes to Redis channel ticks.
  4. Starts one buildcandle(interval, queue) thread per timeframe.
  5. Main thread enters pubsub.listen(), fanning each raw message into all queues.

Main thread — fan-out loop

for data in pubsub.listen():
    raw = data['data']
    for q in tf_queues.values():   # same bytes → every timeframe
        q.put(raw)

OHLCV state machine (one per timeframe thread)

candlesticks[symbol][start_time] = {
    "open":   first last_price seen in the period,
    "high":   running maximum,
    "low":    running minimum,
    "last":   most recent last_price  (= close at boundary),
    "buyer":  latest total_buy_quantity,
    "seller": latest total_sell_quantity,
    "volume": latest volume_traded
}

Candle boundary logic

nexttime advances by interval_minutes each time a boundary is crossed. When exchange_timestamp ≥ nexttime, a copy.deepcopy snapshot is handed to a fresh writeToFile thread. seen_candle_times (a set) prevents the boundary from firing twice. Each timeframe thread is fully independent — a slow symbol in one does not affect others.

writeToFile(candles, start_time, interval_minutes)

Derives channel and filename from interval_minutes via label(). Batches all Redis PUBLISH calls into a single pipeline (one round-trip), then appends all rows to the timeframe-specific CSV in one with open(...) block.

Error isolation: every tick is wrapped in try/except. A malformed tick logs and is skipped — the thread never crashes and one bad tick only produces a slightly imprecise bar, not data loss.

Threading Model

Main Process │ ├── Main Thread [producer] │ └── KiteTicker.connect() — blocks; drives the WebSocket event loop │ ├── Main Thread [consumer] │ └── pubsub.listen() — blocks; feeds messageQueue │ ├── _publish_worker ×4 [producer, daemon] │ └── publish_queue.get() → json.dumps → Redis PUBLISH ticks │ ├── _tick_hash_worker ×1 [producer, daemon] │ └── tick_hash_queue.get() → Redis HSET per field │ ├── buildcandle(1, tf_queues[1]) [consumer, daemon] → 1-min OHLCV state ├── buildcandle(5, tf_queues[5]) [consumer, daemon] → 5-min OHLCV state ├── buildcandle(15, tf_queues[15]) [consumer, daemon] → 15-min OHLCV state ├── buildcandle(30, tf_queues[30]) [consumer, daemon] → 30-min OHLCV state ├── buildcandle(60, tf_queues[60]) [consumer, daemon] → 1-hour OHLCV state │ (add any interval to TIMEFRAMES to spawn another thread) │ └── writeToFile(snapshot, start, tf) ×N [consumer, daemon — one per candle boundary per timeframe] └── deepcopy snapshot → Redis pipeline PUBLISH + timeframe CSV append All workers are daemon threads — killed automatically when the main thread exits. Queues are the only synchronisation mechanism; there are no explicit locks. Each buildcandle thread is fully independent — a crash in one timeframe does not affect others.

Wire Formats

Redis channel: ticks

Published by kite_ticker_producer.py. A JSON-encoded list of tick dicts — one list per on_ticks callback invocation.

[
  {
    "instrument_token":    256265,
    "exchange_timestamp":   "2024-01-15T09:15:03.123456",
    "last_price":           21750.50,
    "total_buy_quantity":   123400,
    "total_sell_quantity":  98700,
    "volume_traded":        4500200,
    "ohlc": {
      "open":  21600.00,
      "high":  21800.00,
      "low":   21550.00,
      "close": 21600.00
    }
  }
]

Redis candle channels (1min_candles, 5min_candles, 1h_candles, …)

Published by min1_ticks.py. One JSON message per symbol per completed bar. Channel name is derived from TIMEFRAMES automatically — same payload shape on every channel.

{
  "timestamp":    "09:15",
  "stockname":    "NIFTY 50",
  "candle_open":  21600.00,
  "candle_high":  21750.50,
  "candle_low":   21580.00,
  "candle_close": 21750.50,
  "candle_buyer": 123400,
  "candle_seller":98700,
  "candle_volume":4500200
}

CSV files (one per timeframe per day)

Appended by writeToFile at each candle boundary. No header row. Filename encodes the timeframe.

Configuration & Setup

Prerequisites

pip install kiteconnect redis pandas requests
redis-server   # must be running locally

~/kite_candles/userdata format

A JSON file read by pandas. Each top-level key is a user profile label used in all scripts.

{
  "YOURNAME": {
    "user":   "AB1234",
    "apikey": "your_api_key_here"
  }
}

Replace YOURNAME with any label. It is used as the pandas column name in all four scripts.

Running the System

Run steps in order. Steps 1–2 are daily; steps 3–4 run each trading day.

Step 1 — Generate instrument list (daily, ~06:00)

python ~/kite_candles/instruments_generator.py

Step 2 — Store today's access token

# Edit set_token.py with today's token, then:
python ~/kite_candles/set_token.py

Step 3 — Start the producer (~08:55)

python ~/kite_candles/kite_ticker_producer.py

Step 4 — Start the 1-minute consumer (~08:55)

python ~/kite_candles/multi-time/1min/min1_ticks.py

Both processes shut themselves down at market close (15:30–15:31). Schedule via crontab:

00 6  * * 1-5  python ~/kite_candles/instruments_generator.py
55 8  * * 1-5  python ~/kite_candles/kite_ticker_producer.py
55 8  * * 1-5  python ~/kite_candles/multi-time/1min/min1_ticks.py

Changing Timeframes / Multiple Candles at Once

Edit the single TIMEFRAMES list at the top of min1_ticks.py. Nothing else changes — channel names, filenames, and thread count are all derived automatically. 

# One timeframe only
TIMEFRAMES = [5]

# Standard multi-timeframe setup
TIMEFRAMES = [1, 5, 15, 30, 60]

# Extended — 2-hour and 6-hour bars included
TIMEFRAMES = [1, 5, 15, 30, 60, 120, 360]
